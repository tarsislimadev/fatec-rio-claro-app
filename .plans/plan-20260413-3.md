# Webapp Plan (Front-end, HTML) - Login/Register + Profile

## Objective
Create a web front-end (HTML-first) for authentication and profile flows:
- Login
- Register
- Profile view
- Profile edit
- Logout

This plan follows the backend Swagger contract from `.plans/plan-20260413-2.md`.

## Scope (v1)
- Public pages: Login, Register
- Protected page: Profile (view + edit)
- Session handling with bearer token
- Basic responsive layout for desktop/mobile
- Accessible HTML forms and validation messages

## Proposed front-end stack
- **HTML5** semantic pages/components
- **CSS3** (modular files or utility approach)
- **Vanilla JavaScript (ES modules)** for API calls and state
- Optional lightweight router (or hash-based routing) for SPA-like navigation

## Information architecture
1. `/login` - user authentication form
2. `/register` - account creation form
3. `/profile` - protected profile page
4. `/` redirect rule:
   - authenticated -> `/profile`
   - unauthenticated -> `/login`

## UI plan by page

### Login page
- Fields: `email`, `password`
- Actions: submit login, navigate to register
- States: idle, loading, field errors, invalid credentials

### Register page
- Fields: `name`, `email`, `password`, `passwordConfirmation`
- Actions: submit register, navigate to login
- States: idle, loading, validation errors, duplicate email

### Profile page (protected)
- View mode: display name/email/role/phone/course/ra/avatar
- Edit mode: update editable fields (`name`, `phone`, `course`, `avatarUrl`)
- Actions: save profile, cancel edit, logout
- States: loading profile, save success, save validation errors, unauthorized

## API integration plan
Use endpoints defined in Swagger plan:
- `POST /auth/register`
- `POST /auth/login`
- `POST /auth/logout`
- `GET /profile/me`
- `PUT /profile/me`

### API client behavior
- Shared `fetch` wrapper with JSON parse and normalized error handling
- Automatic `Authorization` header when token exists
- Global handling for `401`: clear session and redirect to `/login`

## Session and auth gating
- Store token in `localStorage` for v1 (with abstraction in `session.js`)
- On app/page init:
  - If token exists, try `GET /profile/me` to validate session
  - If invalid/expired, clear token and redirect to login
- Block direct access to `/profile` when unauthenticated

## Validation and UX rules
- Client-side validation mirrors backend contract:
  - email format
  - password strength
  - password confirmation match
  - required fields and max lengths
- Show inline field errors + top-level API error summary
- Disable submit buttons during requests
- Provide success feedback after profile save

## Accessibility and quality baseline
- Proper `<label for>` bindings and input `id`s
- Keyboard navigable forms/actions
- ARIA-live region for async success/error messages
- Color contrast compliant styles

## Testing plan (front-end)
1. Manual flow checks:
   - Login success/failure
   - Register success/duplicate email/validation errors
   - Profile load/edit/save/logout
2. Basic automated checks (if test setup exists):
   - Unit tests for validators and session helper
   - Integration tests for API wrapper error mapping

## Incremental milestones
1. **Milestone A:** Static HTML/CSS page skeletons and shared components
2. **Milestone B:** Login/Register wiring with API client + validations
3. **Milestone C:** Protected profile page with edit/save and logout
4. **Milestone D:** Auth gating hardening, accessibility polish, and error handling consistency

## Risks and mitigations
- **Risk:** backend contract drift  
  **Mitigation:** keep API config centralized and aligned to Swagger versioned contract.
- **Risk:** token persistence security tradeoffs in browser  
  **Mitigation:** encapsulate storage and prepare migration path to secure cookie/session strategy if backend supports it.
- **Risk:** inconsistent validation between front-end and back-end  
  **Mitigation:** share documented validation rules and mirror error schema handling.
