# Implementation Plan: Login/Register + Profile (Android)

## Problem and current state
- The app is currently a single `MainActivity` Jetpack Compose entry with placeholder navigation (`Home`, `Favorites`, `Profile`) and a simple greeting screen.
- There is no authentication flow, no profile model/state, no networking layer, and no backend integration yet.
- Priority is to deliver **login/register and profile pages first**, with profile supporting **view, edit, and logout**.
- Backend is not ready, so we will define API contracts and implement against a temporary mock data source while preserving a clean swap to real API later.

## Proposed approach
Build an auth/profile feature slice with clear layers (UI, state, domain, data), introduce a small API contract now, implement a mock repository that matches the contract, and wire navigation so authentication gates profile access. Keep interfaces stable so replacing mock with Retrofit/API later is low-risk.

## Todos
1. **Establish app architecture and feature packages**
   - Create base package structure for `auth` and `profile` with `ui`, `domain`, and `data` separation.
   - Introduce app-level state holder for authentication session and current route.
   - Replace placeholder destination behavior with explicit auth-aware flow.

2. **Define auth/profile API contracts (backend-ready)**
   - Specify request/response models for:
     - `POST /auth/register`
     - `POST /auth/login`
     - `GET /profile/me`
     - `PUT /profile/me`
     - `POST /auth/logout`
   - Define token/session expectations, validation rules, and standard error shape used by app.
   - Keep these contracts as Kotlin models/interfaces in the app data layer so both mock and future real implementation share them.

3. **Implement temporary mock data source**
   - Build an in-memory/mock repository implementing the same contract interfaces.
   - Support success and representative failure scenarios (invalid credentials, duplicate email, validation error, unauthorized).
   - Provide deterministic seeded profile data for predictable UI testing.

4. **Build Login screen**
   - Compose UI with email/password fields, validation, loading, and error display.
   - Submit to auth use case; on success store session state and navigate to profile.
   - Add navigation link to Register screen.

5. **Build Register screen**
   - Compose UI with required fields and validation (name, email, password, confirmation).
   - Submit to register use case; on success auto-login (or redirect to login if decided later) and continue to profile.
   - Display API/validation errors consistently.

6. **Build Profile screen (view + edit + logout)**
   - Fetch and display current profile data.
   - Add editable form with save action, loading, success feedback, and field-level errors.
   - Add logout action that clears session and returns user to login.

7. **Session persistence and auth gating**
   - Add local persistence for auth session/token placeholder (e.g., DataStore/SharedPreferences via abstraction).
   - Restore session on app start.
   - Route users:
     - unauthenticated -> login/register
     - authenticated -> profile

8. **Testing strategy**
   - Add/extend unit tests for auth/profile use cases and mock repository behavior.
   - Add UI tests for critical flows: login success/failure, register validation, profile edit save, logout redirect.
   - Keep test coverage focused on critical first-release paths.

9. **Incremental delivery milestones**
   - Milestone A: Contract + mock + navigation shell
   - Milestone B: Login/Register end-to-end on mock
   - Milestone C: Profile view/edit/logout + session restore
   - Milestone D: Harden errors/tests and prepare swap point for real backend integration

## Notes and considerations
- Because backend is not ready, contract-first design is mandatory to avoid rework.
- Keep mock and real data sources behind one repository interface; only DI binding should change when backend arrives.
- Avoid coupling UI directly to transport models; map DTOs to domain models.
- Preserve existing theme/style patterns and keep all new strings in resources.
